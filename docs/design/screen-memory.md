# 屏幕记忆(Screen Memory)架构设计

状态:设计定稿;L1(帧差静止门)与 L2(存档规格/切条)参数已实测定稿(2026-07-03),待实施
关联:现有 AI 日报管线(`src-tauri/src/ai/`)、`screenshot_dedup_map`(迁移 v35)、标题守卫去重(`ai/dedup.rs`)

---

## 1. 目标与约束

本设计统一目前分散的 AI 子系统(MobileNet 去重、OCR 规划、VLM 日报、Chat 规划),
面向**内置 Chat Bot** 这一最终消费端,回答如下类型的问题:

> 上周我在 Cursor 用了多久?我今天专注度如何?我什么时候购买的 XX?我下午在看什么视频?

硬约束:

1. **丢失有界**——信息完整性与算力成本权衡,不追求零丢失;可接受有界、可预期的丢失,但不容随机的永久漏。
2. **隐私红线**——截图数据永不出设备。

旧方案的失败模式(为什么推倒):

| 旧做法 | 失败点 |
|---|---|
| 每晚 VLM 描述去重后全部帧(~120 张/天) | 贵(GPU 十几分钟,CPU 不可用);没被抽中的帧信息永久丢失 |
| MobileNet(ImageNet 特征)语义去重 | 对文字瞎:同店铺不同商品页/同编辑器不同文件被误合并,需要标题守卫兜底 |
| 只上 OCR | 纯视觉页面(设计稿/照片/视频画面)必然漏 |
| 去重 = 删帧 | 必然丢信息;不去重则下游成本爆炸 |

│  ├─ 统计 SQL(时长/占比类问题)                     │

│  ├─ CLIP 语义搜图(看过什么,文本→图)              │
│  ├─ 簇描述缓存(在干什么)                         │

核心思想:**级联漏斗**。每一级只把"上一级解释不了的帧"往下送;越贵的工具见到的帧越少,
且见到的必然值得。没有任何一级执行"删除"——只有"附着"(attach),
每帧的时间戳与归属永远可查。

## 3. 五级瀑布详解

以典型日为例:8h 录制 × 30s 间隔 ≈ **960 帧**。

### L0 元数据(免费,100% 保留)

活动行(app、窗口标题、起止时间)本来就在主库。它是所有下游层的分组键,
也是隐私跳过时段(不截图的应用)的唯一信息来源——这类时段本来就只该有元数据。

### L1 静止门——帧差法(~1ms 级,拦 ~5-15%)

**定位:静止检测器,不是内容检测器**——只回答"这帧确定没变吗",拿不准一律放行。
内容级判断全部下放 L3(文本层,内容可见)。误放行的代价被 OCR 单价(0.13s)封顶,
全天全量误放行 ≈ 2 分钟计算;误附着则丢内容——两者代价不对称,故阈值偏保守(但按 §10 权衡,非零漏)。

**执行位置:抓拍管线内、JPEG 编码之前——附着的帧不写盘**(用户裁定)。
锚点帧的时间范围天然覆盖被附着的间隔;静止时段零新文件,磁盘/IO/retention churn 同步减免。

```
状态:锚点缩略(36KB/屏)+ 挂起帧(内存,≤1 帧,~0.3MB)
1. 新帧缩到 256×144 灰度,逐像素 |cur − anchor| > τ(=12)记变化,fraction=占比
2. fraction < f          → 附着:不写盘,锚点不动,记 dwell
3. fraction ≥ f          → 挂起(帧留内存,不落盘),等下一帧:
     下一帧 vs 锚点 < f   → 瞬态弹层(IME 候选框/通知横幅/菜单)→ 丢弃挂起帧
     下一帧 vs 锚点 ≥ f   → 持久变化确认 → 挂起帧补写盘(时间戳保留),锚点 := 该帧
4. 兜底阀:连续附着 K=4 帧 → 第 5 帧无条件走确认流程
   输出变化块掩码传给 L2(区域 OCR:裁条顺带解 Vision 相对图高地板)
```

- **持久性确认(一帧延迟)**:瞬态弹层(IME/通知/菜单)与真实编辑在变化面积上不可分,
  可分的是时间性——弹层撑不过一个采样间隔。越阈帧挂起一帧,下一帧回落即丢弃。
- **跟锚点比而非上一帧比**:打字类缓变相对锚点单调累积,必然越阈。
- **运营点**:f = 0.10% 单值,全分辨率通用(标定与召回数据见 §10 评测资产的脚本输出)。
  漏检方向安全:打字级小变化由 L3 行级并集吸收;最小面积原地替换由 K 阀封顶。
- **哈希族已证伪,勿回退**:网格均值/梯度类摘要(dHash 等)对同密度文本替换结构性
  失明,任何阈值无解。dHash16 仅保留为存储指纹(跨天近似查重),不守门。
- 已知盲区(均由 K 阀封顶):等亮度纯色变、低于 τ 的低对比变化、重复图案文本混叠
  (终端日志/表格滚动后像素对齐,丢的只是重复行内时间戳级差异)。

资源:帧差 ~1ms/帧,全天 ≈ 1-2s CPU,常驻 36KB/屏。
远期:插电常驻改持续捕获会话后,本层由合成器 dirty rects(SCK/DXGI)免费替代。

**配套的采集层改动:快门与时长记账解耦**

快门(截不截图)的活跃判据 = **屏幕在变**(L1 同款传感器);键鼠空闲只用于时长记账
(挂机封会话、不计时长)。两者不解耦,看视频/盯编译这类零输入时段会整段断供。

- 真挂机(屏幕静止):帧全部附着,不写盘,成本 ≈ 0;
- 被动观看(屏幕在变):帧照常进入 L2/L4,并计入使用时长。

过渡实现已上线(`service.rs::screen_probe_active`,挂机判定第二信号,限频 20s、
截图功能关闭时停用);P1 时探测器并入 L1 静止门,同一传感器不留两套。

### L2 文本层——OCR 逐字识别(0.13s/帧,全量)

所有通过 L1 的帧(~350 帧/天)全部过系统 OCR(macOS Vision framework / Windows OCR)。
实测(用户机器):1280×720 与全分辨率速度相同 ~130ms/帧;350 帧 ≈ **45 秒/天**总计算量,
便宜到不需要抽样。产出逐字文本,进 FTS 索引(trigram tokenizer,支持中日文子串)。

设计要点:**OCR 在去重之前**——先全录,再折叠。旧方案"先去重再分析"的信息丢失由此根除。

**输入分辨率结论(POC 实测,`scripts/poc/l2-ocr-accuracy-poc.py` + `scripts/poc/ocr-cli.swift`)**:

- Vision 两条地板:文字**相对图高 ≲1%** 整行检测不到(**切条可解**:横切 3-4 条、重叠 48px);
  **原生渲染 ≲13px** 的细笔画识别不动(**无解**:无损 PNG 同样烂,整图放大/插值救不回,
  实测 16px 切条 99%,13px 切条仅 27-33%)。JPEG q80 无锅(98.6% vs 无损 99.2%)。
- **屏幕 DPI 决定生死**:2× Retina 屏 UI 文字 26px+ 物理,现有 1280 存档缩后是"高墨量 13px",
  99.7% 可读——**现状即可用**;1× 屏(QHD/1080p@100%)正文 14-16px 物理骑在地板上,
  **1280/q80 存档为不可逆毁灭**(实测存档图 225 乱码字符),历史档案无法回填。
- 一般人字号分布:界面 chrome 12-13px、正文 14-16px、代码 12-14(逻辑px)——
  1× 屏的 chrome 永久放弃(有 L0 标题兜底),正文靠原生帧 + 切条挽救。
- **存档规格(已定案)**:`存档宽度 = 屏幕逻辑宽度(native ÷ scale),上限 2880;
  JPEG q85;Lanczos3 滤镜`(替换现状 1280/q80/Triangle)。
  - 1× 屏(QHD 2560)→ 存原生 2560,~300KB/张:14-16px 正文以原生锐利像素进 OCR;
  - 2× 屏(MacBook 2880)→ 存 1440,~180KB/张:26px 物理文字缩成"高墨量 13px",
    实测 99.7% 可读,且恰为逻辑分辨率、1:1 观看不糊;
  - 4K@100%(罕见)超上限截到 2880,小字接受损失并在文档声明。
  - 磁盘账:单张 2-4×,但 L1 静止帧不落盘抵消一部分,净增约 2-3×,retention 管上限。
  - 历史档案(1× 屏来源)无法回填文本,从新规格起累积。
- L1 输出的变化块掩码 → **区域 OCR**(只 OCR 变化条带):省时 + 裁剪天然抬高相对图高。

### L3 会话折叠——标题定界 + 行级并集(μs 级,真正的去重层)

**去重的对象是帧,合并的是文字**:同一次阅读/编辑的 N 帧折叠成一条会话记忆,
文本取**行级并集**,一行不丢。这是全瀑布里唯一"看得见内容"的去重,
原地替换、大幅滚动等像素层盲区全部在此兜住。

```
会话边界 = (app, 标准化窗口标题) + 时间连续(间隔 ≤ 5min)
会话内每帧三态判定:
  ① 新行占比 ≈ 0(全部已在会话文本里)→ 附着(回看/静止)
  ② 有新行 → 并入:新行追加(记本帧时间戳=行级留痕),帧记为会话成员
  ③ 标题变 / 时间断裂 → 封会话,开新会话
```

- **simhash 降级为帧级预筛**(与上一帧文本几乎相同 → 跳过行级比对),
  **不再决定会话边界**——同窗口大幅滚动两视口文本几乎零重叠(实测真实案例:
  claude.ai 长对话滚动,86 行 + 58 行、仅 16 行重叠),按 simhash 切会话会把
  一次阅读碎成多条;按"选代表帧"去重则白丢 42 行独有内容。
- 行级留痕:每行记首次出现帧 → 搜索命中可给出精确到帧的证据卡。
- SPA 软分割(同标题内容全换):连续多帧与会话文本零重叠 → 会话内标记分割点,
  P2 打磨,不影响信息完整性,只影响时间线叙事粒度。
- **旧管线警示**:MobileNet 语义去重对"同标题滚动/替换"必然误合并
  (同版面 → 嵌入相似度 >0.95,标题守卫不拦同标题)——P3 换血前此坑常在,
  过渡期可将阈值临时上调至 0.97 收敛。
- 写码/文档/网页浏览等文字型工作 ~80% 的帧在本层终结,不进 L4/L5。

### L4 视觉门——CLIP 嵌入 + 新颖度判定(~ms 级)

**只处理文本解释不了的帧**:OCR 读不出几个字(设计稿/照片/视频/游戏),
或文字没变但画面剧变。

1. 用 CLIP 族模型(MobileCLIP-S / SigLIP 类,ONNX 本地,几 ms/帧)算图像嵌入;
2. 与当日已有**视觉簇**(cluster)的代表向量比余弦相似度:
   - ≥ 簇半径阈值 → **附着**该簇(记时间戳,写留痕),不进 L5;
   - 全部不像 → 判定为**新视觉场景**,建新簇,升级 L5。

为什么换掉 MobileNet:ImageNet 特征只认版面纹理不认内容;CLIP 特征与文本语义对齐,
一份嵌入三个用途——新颖度门、Chat 的文本→图检索、日报的时间线聚簇。
标题守卫(`ai/dedup.rs`)保留为附着前的保险:两帧标题都非空且不同 → 禁止附着。

### L5 理解层——VLM 描述(2-10s/帧,5-30 张/天)

只看两种帧:

1. **新簇首帧**——每个前所未见的视觉场景,保证被看一次。描述缓存为簇代表描述;
2. **长驻重采样**——存活超过 N 分钟(建议 20-30min)的簇,每 N 分钟补看一帧,
   捕捉"缓慢演变"(改了两小时的设计稿、看了一下午的长视频)。

调度:全部在消化 worker 内跑(插电/空闲优先),设**日预算帽**(建议 40 张/天,
优先级:新簇 > 重采样 > 积压回填)。Chat 提问命中未描述簇时**懒富集**:
现场看图(≤8 张),结果写回缓存,同一帧终生只看一次。

成本公式由此改变:**O(帧数) → O(新场景数 + 长驻小时数)**。
写码日 ≈ 0-5 次 VLM 调用;设计/剪辑日 ≈ 20-40 次;而非无差别 120 张。

## 4. 信息丢失界

三档丢失,每档有明确的界:

| 档 | 什么帧 | 丢什么 | 保障 |
|---|---|---|---|
| 零丢失 | L1 静止帧(帧差<f=0.10%) | 无(像素未变;1440 档最小面积替换有 4% 漏检) | 持久性确认 + K=4 连附着强制落盘阀(≤2.5min) |
| 逐字保留 | L3 会话成员帧 | 版面装饰 | 行级并集一行不丢,FTS 可搜,行级留痕到帧 |
| 有界丢失 | L4 簇内成员 | 簇半径内的渐变细节 | 簇代表描述 + 全部时间戳 + 重采样封顶 |

按信息类型看兜底通道:

| 信息类型 | 兜底通道 | 漏? |
|---|---|---|
| 屏幕文字(商品/聊天/代码/金额/字幕) | L2 OCR 逐字全量 | 不漏,逐字忠实 |
| 纯视觉内容(设计稿/照片/视频画面) | L4 保证建簇 + L5 保证首帧被看 | 不漏,只是"晚看" |
| 被动观看时段(视频/编译,零键鼠输入) | 屏幕活跃探测保持 roll 补帧(已实施);P1 后由 L1 裁决 | 不漏(修复前:开场一帧后断供) |
| 被折叠/附着的帧 | 留痕表(`member → rep` + 时间戳) | 不漏,时间段可还原 |
| 截图被 retention 删除后 | 文本与描述留在 memory.sqlite | 图没了字还在 |
| 隐私应用跳过截图的时段 | L0 活动行 | 只有画面,本该如此 |

## 5. 数据模型:memory.sqlite(独立数据库)

独立于主库(activities.sqlite)的第二个数据库文件。理由:
用户可独立删除/迁移这份资产;避开主库单连接(tokio_rusqlite)的写争用;
物理隔离保证永不进云同步。

```sql
-- 每个保留帧一条(通过 L1 的帧)
CREATE TABLE frames (
    path        TEXT PRIMARY KEY,   -- 截图相对路径(与主库对齐)
    ts          TEXT NOT NULL,      -- 拍摄时刻
    local_date  TEXT NOT NULL,
    app_id      TEXT, title TEXT,   -- L0 快照
    phash       INTEGER,            -- L1
    ocr_state   INTEGER NOT NULL DEFAULT 0,  -- 0待 1完 2失败
    session_id  INTEGER,            -- L3 文本会话归属
    cluster_id  INTEGER             -- L4 视觉簇归属(NULL=文字帧)
);

-- L3 文本会话(折叠后的检索单元)
CREATE TABLE text_sessions (
    id INTEGER PRIMARY KEY,
    local_date TEXT NOT NULL,
    started_ts TEXT NOT NULL, ended_ts TEXT NOT NULL,
    app_id TEXT, title TEXT,
    simhash INTEGER,
    text TEXT NOT NULL              -- 会话合并文本(增量拼接)
);
CREATE VIRTUAL TABLE text_sessions_fts USING fts5(
    text, content='text_sessions', content_rowid='id',
    tokenize='trigram'              -- 中日文子串搜索
);

-- L4 视觉簇
CREATE TABLE clusters (
    id INTEGER PRIMARY KEY,
    local_date TEXT NOT NULL,
    rep_path TEXT NOT NULL,         -- 簇代表帧
    embedding BLOB NOT NULL,        -- 代表向量(f32 数组)
    description TEXT,               -- L5 缓存(NULL=未描述)
    described_at TEXT
);

-- 帧嵌入(供 CLIP 文本→图检索;仅 L4 处理过的帧)
CREATE TABLE frame_embeddings (
    path TEXT PRIMARY KEY,
    embedding BLOB NOT NULL
);

-- 附着留痕(L1/L3/L4 统一账本;迁移自主库 screenshot_dedup_map)
CREATE TABLE attach_map (
    member_path TEXT PRIMARY KEY,
    rep_path    TEXT NOT NULL,
    level       INTEGER NOT NULL,   -- 1=pHash 3=text 4=cluster
    local_date  TEXT NOT NULL,
    created_at  TEXT NOT NULL
);
```

嵌入检索规模:每天 L4 帧数 × 512 维 f32,一年 ≈ 几万行 × 2KB,
暴力扫描 + 余弦即可(<100ms),不需要向量数据库。

## 6. 消化 worker 生命周期

复用已定的 OCR worker 方案,扩展为全瀑布:

- **形态**:独立子进程(`hindsight --digest-worker`),spawn → 加载模型 → 消化积压 → 退出。
  OCR 模型编译为 OS 级缓存(实测首次 12s 一次性,之后新进程 ~200ms 冷启);
  CLIP ONNX 常内存 ~100MB、VLM 按需由 EngineSupervisor 管理——都只挂在 worker 存续期。
- **两档模式**(设置项,仅截图开启时可见):
  - **批量(默认)**:触发时机 = 生成日报前 / 系统空闲 / 每日定时。一次消化完积压即退出;
  - **插电常驻(opt-in)**:接电源时驻留,准实时消化 + 预富集(提前跑 L5 新簇);
    拔电即退,电池上永不常驻。
- **失败语义**:任何一级失败只降级不阻塞——OCR 失败的帧仍有 L0/L1;
  VLM 失败的簇留 `description=NULL` 等懒富集。`ocr_state=2` 可重试。

## 7. Chat 大脑(唯一消费端)

tool-calling 循环,工具面(每类问题都有廉价通道):

| 工具 | 数据源 | 回答什么 |
|---|---|---|
| `query_stats` | 主库 SQL | 时长/占比/趋势("Cursor 用了多久") |
| `search_text` | text_sessions_fts | 逐字内容("什么时候买的 XX") |
| `search_visual` | frame_embeddings × CLIP 文本编码 | 语义找图("看机械键盘的页面") |
| `get_timeline` | clusters + text_sessions 按时间 | 某时段在干什么(日报骨架) |
| `look_at_images` | VLM 现场看图(≤8 张) | 兜底精读;结果写回 clusters.description |

回答携带**证据卡**:命中的时间段、截图缩略图、来源(逐字/描述/统计),点击跳转时间线。

### 报告并轨

日报 = 每晚定时触发的一次固定模板 Chat 查询(`get_timeline` + `query_stats` → 叙事),
周报/月报同理。现有 step1(夜跑 VLM 描述全部帧)/step2 管线在 P3 下线。
用户手动提问与系统自动报告走同一条代码路径。

## 10. 参数表(实测定案 + 待拍板)

**L1 静止门(已实测定案,`scripts/poc/l1-phash-poc.py --fd`)**

| 参数 | 值 | 依据 |
|---|---|---|
| 执行位置 | 抓拍管线内,编码前;**附着帧不写盘** | 用户裁定;静止时段零新文件 |
| 缩略尺寸 | 256×144 灰度 | 半个字/缩略像素,替换必可见;36KB/屏 |
| 像素阈值 τ | 12 / 255 | 滤 JPEG 纹波/字体平滑;真实变化 30-150 级 |
| 占比阈值 f | **0.10% 单值**(放宽,非 100% 召回) | 675/1080 档召回≈100%,1440 档最小正类 75-85% |
| 持久性确认 | 越阈帧挂起 1 帧,下一帧变化消失→丢弃 | IME/通知横幅等瞬态弹层 100% 拦截 |
| 兜底阀 K | 连续附着 4 帧强制走确认 | 漏检上确界 ≈ 2.5min 必落盘一帧 |
| 锚点语义 | 与上一保留帧比,附着不更新锚点 | 缓变累积必越阈 |
| 实测 | 抓拍路径 0.85ms/帧;真实流附着率 ~6% | 全天 ~1-2s CPU;挂起帧内存 ~0.3MB |

**L2 OCR(已实测定案)**

| 参数 | 值 | 依据 |
|---|---|---|
| 切条 | 3-4 横条,重叠 48px | 解 Vision ~1% 相对图高检测地板 |
| 输入下限 | 原生渲染 ≥16px 物理文字 | 13px 即使无损也仅 27-33%;16px 切条 99% |
| 识别质量评法 | 句子还原率(逐字地面真值) | 字符数/字多者为基准均已证伪 |
| 存档规格 | **宽 = 逻辑宽度(≤2880),q85,Lanczos3** | 1×屏原生锐利、2×屏高墨量缩放,两个 OCR 可用域 |

**L3 会话折叠(设计定案,阈值待实装标定)**

| 参数 | 建议 | 说明 |
|---|---|---|
| 会话间隔上限 | 5 min | 同标题断流即封会话 |
| 行有效长度 | ≥6 字符(标准化后) | 滤图标/单字噪音 |
| simhash 预筛 | 距离 ≤3 跳过行级比对 | 仅加速,不定边界 |
| SPA 软分割 | 连续 3 帧与会话文本零重叠 | P2 |

**L4/L5(沿用原设计,CLIP 选型与簇半径待 P3 POC)**:簇半径 0.90、重采样 25min、
日预算 40 张、CLIP 取 MobileCLIP-S 级 ONNX(中文文本编码质量待验)。

全部参数落 `ai` 配置节,调试页暴露;附着率/会话数/簇数/VLM 用量在调试页可观测。
评测资产:`scripts/poc/l1-phash-poc.py`(--pr 标注集评测 / --fd 帧差标定+资源基准)、
`scripts/poc/l1-dedup-eval.py`(真实截图流离线评测+全帧可视化报告,2026-07-05 首跑:
活跃开发日附着率 3.5%,机制全部按预期工作)、
`scripts/poc/l2-ocr-accuracy-poc.py`、`scripts/poc/ocr-cli.swift`;样张 `scripts/poc/output/l1|l2/`(已 gitignore)。
